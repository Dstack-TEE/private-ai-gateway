# ACI Test Vectors

Deterministic, byte-exact vectors for every digest, signature, and sealing
construction in the [ACI Spec](aci.md). An implementation that reproduces
every value below has its binding logic right — which is where independent
implementations diverge in practice.

There is no canonical JSON in ACI (spec §3). Artifacts — the keyset, the
session document, the receipt payload — are verified as **served bytes**: the
producer serializes once, and a verifier hashes and signature-checks exactly
the bytes it fetched. The artifact bytes below are the bytes the reference
implementation serves for this content; a verifier must reproduce every
digest and verification result from them without re-serializing. The only
payloads a verifier constructs itself are the two fixed byte templates: the
attestation statement (§2) and the E2EE AAD (§5).

Every value is regenerated by [`tools/gen-vectors.py`](tools/gen-vectors.py)
(python3 + `cryptography`), an implementation independent of the reference,
which self-checks against the constants below. The reference implementation
pins the same constants in `tests/spec_vectors.rs`.

Conventions: all artifact bytes are ASCII, compact JSON (no whitespace).
Digests are `sha256:<lowercase-hex>`; `report_data` and `*_sha256` fields are
bare hex; `*_b64` fields are standard base64 with padding; keys and
signatures are hex with no `0x` prefix (spec §3).

## Fixed keys

Private keys are 32-byte seeds. Nothing here is random: the E2EE vectors pin
the ephemeral keys and GCM nonces that a real client draws fresh per sealed
unit (§7.1).

| Key | Seed (private) | Public key (hex) |
| --- | --- | --- |
| receipt `receipt-1` (Ed25519) | 32 × `0x02` | `8139770ea87d175f56a35466c34c7ecccb8d8a91b4ee37a25df60f5b8fc9b394` |
| service E2EE `e2ee-1` (X25519) | 32 × `0x03` | `5dfedd3b6bd47f6fa28ee15d969d5bb0ea53774d488bdaf9df1c6e0124b3ef22` |
| client E2EE (X25519) | 32 × `0x04` | `ac01b2209e86354fb853237b5de0f4fab13c7fcbf433a61c019369617fecf10b` |
| request ephemeral (X25519) | 32 × `0x05` | `50a61409b1ddd0325e9b16b700e719e9772c07000b1bd7786e907c653d20495d` |
| response ephemeral (X25519) | 32 × `0x06` | `f5b2d6e60f9477e310c2982daaa6c9136c108a1777c5947e448fa37d68174557` |
| SSE-event ephemeral (X25519) | 32 × `0x07` | `13be4feaeaf204c7fd3358fc9c00721881d174278128227ec674f37f7fe97b6d` |

The TLS SPKI (`c0` × 32) and the session channel-binding SPKI (`d1` × 32) are
placeholder digests; no certificate exists behind them.

## 5. E2EE v3 sealing (spec §7)

One construction seals everything (§7.1):

```text
shared_secret     = X25519(ephemeral_private_key, recipient_public_key)
key               = HKDF-SHA256(salt = <absent>, ikm = shared_secret, info = UTF-8(context), length = 32)
request_aad       = UTF-8(context) || 0x00 || UTF-8(model) || 0x00 || UTF-8(client_key_hex)
response_aad      = UTF-8(context) || 0x00 || UTF-8(model)
ciphertext || tag = AES-256-GCM(key, gcm_nonce, plaintext, aad)
sealed            = ephemeral_public_key (32) || gcm_nonce (12) || ciphertext || tag (16)
```

The envelope `model` is `demo-model` in both directions.
